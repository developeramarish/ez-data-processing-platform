## סקירה כללית

### מה זה EZ Platform?

פלטפורמת EZ היא מערכת לעיבוד אוטומטי של נתונים המאפשרת:
- 📁 **גילוי קבצים אוטומטי** ממקורות שונים (תיקיות, SFTP, FTP, HTTP, Kafka)
- 🔄 **המרת פורמטים** (CSV, JSON, XML, Excel)
- ✅ **אימות נתונים** לפי סכמה שהוגדרה מראש
- 📤 **שליחה ליעדים מרובים** (תיקיות, Kafka topics)
- 🚨 **ניהול רשומות שגויות** וטיפול בשגיאות
- 📊 **מעקב ומדדים** בזמן אמת

### למי המערכת מיועדת?

- **מנהלי נתונים**: הגדרה וניהול של מקורות נתונים
- **אנליסטים**: הגדרת סכמות ותהליכי אימות
- **מפעילי מערכת**: מעקב אחר עיבוד ותזמון
- **מנהלי איכות**: זיהוי וטיפול בשגיאות

## ניהול מקורות נתונים

### מה זה מקור נתונים?

מקור נתונים מגדיר:
- **מהיכן** לקחת קבצים (תיקייה, שרת SFTP, HTTP API, וכו')
- **איזה פורמט** הקבצים (CSV, JSON, XML, Excel)
- **איך לאמת** את הנתונים (JSON Schema)
- **לאן לשלוח** את הנתונים המעובדים
- **מתי לעבד** (תזמון אוטומטי או ידני)

---

### יצירת מקור נתונים חדש

#### שלב 1: מידע בסיסי

1. לחץ על "מקורות נתונים" בתפריט הצדדי
2. לחץ על כפתור "הוסף מקור נתונים"
3. מלא פרטים בסיסיים:

**שם מקור הנתונים:**
- שם מזהה ייחודי (לדוגמה: "נתוני מכירות חודשיים")
- חובה, 2-100 תווים

**שם הספק:**
- שם הגוף המספק את הנתונים (לדוגמה: "חברת ABC")
- חובה, 2-50 תווים

**קטגוריה:**
- בחר קטגוריה מהרשימה (מכירות, כספים, מלאי, וכו')
- רשימת הקטגוריות מנוהלת על ידי מנהל המערכת
- אם אין קטגוריות זמינות, יופיע קישור ליצירת קטגוריה חדשה

**תקופת שמירה:**
- כמה ימים לשמור קבצים מעובדים (ברירת מחדל: 30 יום)

**תיאור:**
- הסבר מפורט על מקור הנתונים (אופציונלי, עד 500 תווים)

**סטטוס:**
- פעיל / לא פעיל
- רק מקורות פעילים מעובדים על ידי המערכת

4. לחץ "המשך" למעבר לכרטיסייה הבאה

#### שלב 2: הגדרות חיבור

בחר סוג חיבור:

**תיקיה מקומית (Local):**
- נתיב מלא לתיקיה (לדוגמה: `/data/uploads`)
- המערכת תסרוק קבצים חדשים בתיקייה

**SFTP:**
- כתובת שרת (לדוגמה: `sftp.example.com`)
- שם משתמש וסיסמה
- נתיב בשרת
- יציאה (ברירת מחדל: 22)

**FTP:**
- כתובת שרת
- שם משתמש וסיסמה (אופציונלי)
- נתיב בשרת
- יציאה (ברירת מחדל: 21)

**HTTP API:**
- כתובת URL (לדוגמה: `https://api.example.com/data`)
- Headers (אופציונלי)
- שיטת אימות (API Key, Bearer Token)

**Kafka:**
- כתובת Broker (לדוגמה: `kafka.example.com:9092`)
- שם Topic
- Consumer Group
- פרוטוקול אבטחה (PLAINTEXT, SASL_SSL, וכו')

**בדיקת חיבור:**
- לחץ "בדוק חיבור" לאימות שהחיבור עובד
- סטטוס יופיע: ✅ מוצלח או ❌ נכשל עם הודעת שגיאה

#### שלב 3: הגדרות קובץ

**פורמט קובץ:**
- CSV, JSON, XML, Excel
- המערכת תמיר אוטומטית לפורמט אחיד

**הגדרות CSV:**
- **מפריד (Delimiter)**: פסיק, נקודה-פסיק, Tab, צינור
- **קידוד (Encoding)**: UTF-8 (מומלץ), Windows-1255, ISO-8859-8
- **שורת כותרת**: האם השורה הראשונה מכילה שמות עמודות

**תבנית שם קובץ (Pattern):**
- Regex לסינון קבצים (לדוגמה: `sales-.*\.csv`)
- רק קבצים התואמים לתבנית יעובדו

**גודל מקסימלי:**
- הגבלת גודל קובץ (MB)
- ברירת מחדל: 100MB

#### שלב 4: הגדרת Schema

**מה זה Schema?**
- JSON Schema המגדיר את מבנה הנתונים הצפוי
- כללי אימות לכל שדה
- שדות חובה ואופציונליים

**אפשרויות:**

**1. בחירת Schema קיים:**
- בחר מרשימת Schemas שהוגדרו במערכת
- מתאים כאשר יש כבר Schema מוכן

**2. יצירת Schema חדש:**
- לחץ "צור Schema חדש"
- השתמש בעורך החזותי
- הוסף שדות ואימותים
- שמור Schema

**3. ללא Schema:**
- המערכת תקבל את כל הנתונים ללא אימות
- לא מומלץ לסביבת ייצור

**עורך Schema החזותי:**
- הוסף שדות: שם, סוג (מחרוזת, מספר, תאריך, וכו')
- הגדר אימותים: אורך מינימלי/מקסימלי, תבנית Regex, טווח ערכים
- סמן שדות חובה
- הוסף תיאור לכל שדה

**תבניות Schema:**
- לקוח ישראלי (עם ת\"ז, טלפון, כתובת)
- עסקה פיננסית
- קטלוג מוצרים
- רשומת עובד
- חשבונית מע\"מ

#### שלב 5: תזמון

**תדירות עיבוד:**

**ידני:**
- המערכת לא תעבד אוטומטית
- ניתן להפעיל ידנית מרשימת מקורות הנתונים

**מתוזמן:**
- **כל דקה**: בדיקה כל דקה
- **כל 5 דקות**: בדיקה כל 5 דקות
- **כל 15 דקות**: בדיקה כל 15 דקות
- **כל 30 דקות**: בדיקה כל חצי שעה
- **כל שעה**: בדיקה כל שעה
- **יומי**: פעם ביום בשעה קבועה
- **שבועי**: פעם בשבוע ביום ובשעה קבועים
- **חודשי**: פעם בחודש ביום ובשעה קבועים
- **מותאם אישית (Cron)**: הגדרת ביטוי Cron מותאם

**דוגמאות Cron:**
```
*/5 * * * *    - כל 5 דקות
0 9 * * *      - כל יום ב-9:00
0 0 * * 0      - כל יום ראשון ב-00:00
0 9 1 * *      - ב-1 לחודש ב-9:00
```

**חלון זמן:**
- הגדר טווח שעות לעיבוד (לדוגמה: 08:00-18:00)
- מחוץ לחלון, המערכת לא תעבד קבצים

#### שלב 6: כללי אימות

**פעולה בעת שגיאת אימות:**

**המשך עיבוד:**
- רשומות תקינות עוברות הלאה
- רשומות שגויות נשלחות ל"רשומות לא תקינות"
- מומלץ לרוב המקרים

**הפסק עיבוד:**
- כל שגיאת אימות עוצרת את העיבוד
- כל הקובץ נדחה
- מתאים לנתונים קריטיים

**אחוז מקסימלי של שגיאות:**
- הגדר סף (לדוגמה: 10%)
- אם אחוז השגיאות גבוה מהסף, העיבוד נעצר

#### שלב 7: התראות

**התראות על:**
- קובץ חדש התגלה
- אימות הושלם בהצלחה
- שגיאות באימות
- שגיאות בעיבוד
- פלט הושלם

**ערוצי התראה:**
- Email (פיצ'ר עתידי)
- Webhook (פיצ'ר עתידי)
- UI notifications ✅

#### שלב 8: יעדי פלט

**הגדר לאן לשלוח נתונים מעובדים:**

**תיקייה (Folder):**
- נתיב יעד
- פורמט: JSON, CSV, XML
- שם קובץ (תבנית עם placeholders)

**Kafka Topic:**
- שרת Kafka
- שם Topic
- פורמט הודעה: JSON
- כולל Invalid Records: כן/לא

**יעדים מרובים:**
- ניתן להוסיף מספר יעדי פלט
- לחץ "הוסף יעד" להוספת יעד נוסף
- לדוגמה: שליחה לתיקייה + Kafka Topic

#### שלב 9: שמירה

1. בדוק שכל השדות הנדרשים מולאו
2. לחץ "צור" (או "עדכן" במצב עריכה)
3. מקור הנתונים יופיע ברשימה
4. המערכת תתחיל לעבד קבצים לפי התזמון שהוגדר

2. המערכת תתחיל עיבוד מיידי
3. הודעה תופיע: "הפעלה ידנית בוצעה בהצלחה"

## הגדרת סכמות

### סקירת Schema

JSON Schema מגדיר:
- **מבנה הנתונים**: אילו שדות צריכים להיות
- **סוג הנתונים**: מחרוזת, מספר, תאריך, וכו'
- **כללי אימות**: אורך, תבנית, טווח ערכים
- **שדות חובה**: אילו שדות חייבים להופיע

---

### יצירת Schema חדש

#### דרך 1: תבנית מוכנה

1. עבור ל"ניהול Schema"
2. לחץ "השתמש בתבנית"
3. בחר תבנית:
   - **לקוח ישראלי**: שדות לקוח עם ת\"ז, טלפון, כתובת
   - **עסקה פיננסית**: סכום, תאריך, סוג עסקה
   - **קטלוג מוצרים**: שם, מחיר, מלאי
   - **רשומת עובד**: שם, תפקיד, משכורת
   - **חשבונית מע\"מ**: מספר, תאריך, סכום, מע\"מ
4. התאם אישית לפי הצורך
5. שמור Schema

#### דרך 2: עורך חזותי

1. לחץ "צור Schema חדש"
2. מלא פרטים בסיסיים:
   - **שם Schema**: שם טכני באנגלית (lowercase, underscores)
   - **שם תצוגה**: שם בעברית
   - **תיאור**: הסבר מה Schema זה מאמת

3. הוסף שדות:
   - לחץ "הוסף שדה"
   - **שם שדה**: שם באנגלית (לדוגמה: `customer_id`)
   - **שם תצוגה**: שם בעברית (לדוגמה: "מספר לקוח")
   - **סוג**: בחר מהרשימה
   - **חובה**: סמן אם השדה הוא חובה

4. הגדר אימותים לכל שדה:

**עבור מחרוזת (String):**
- אורך מינימלי/מקסימלי
- תבנית Regex (לדוגמה: ת\"ז ישראלית, מספר טלפון)
- פורמט (email, uri, date-time)

**עבור מספר (Number):**
- ערך מינימלי/מקסימלי
- כפולה של (multipleOf)

**עבור תאריך:**
- פורמט: date, date-time, time
- טווח תאריכים

**עבור boolean:**
- ערכים: true/false

5. שמור Schema
6. ניתן לקשר Schema למקור נתונים

---

### עריכת Schema קיים

1. ברשימת Schemas, לחץ "ערוך"
2. ערוך שדות או אימותים
3. שמור שינויים

**⚠️ שים לב:**
- שינוי Schema משפיע על קבצים עתידיים בלבד
- קבצים שכבר עובדו לא מושפעים

## תזמון ועיבוד

### מעגל החיים של קובץ

```
1. גילוי (FileDiscovery)
   ↓
2. המרת פורמט (FileProcessor)
   ↓
3. אימות (Validation)
   ↓
4. פלט (Output)
```

**זמן עיבוד ממוצע:**
- קובץ קטן (<1000 רשומות): 1-3 שניות
- קובץ בינוני (1000-5000 רשומות): 5-15 שניות
- קובץ גדול (5000-10000 רשומות): 20-60 שניות

---

### מעקב אחר עיבוד

**ברשימת מקורות נתונים:**
- עמודת "קבצים": מספר קבצים שעובדו
- עמודת "תזמון": תדירות העיבוד
- כפתור ⚡: הפעלה ידנית

**בדשבורד:**
- גרפים של קבצים שעובדו לאורך זמן
- מספר רשומות מעובדות
- אחוז שגיאות

---

### הפעלה ידנית

**מתי להשתמש:**
- בדיקת מקור נתונים חדש
- עיבוד מיידי של קבצים דחופים
- לאחר תיקון שגיאות

**איך:**
1. מצא את מקור הנתונים ברשימה
2. לחץ כפתור ⚡ (ברק)
3. המערכת מתחילה עיבוד מיידי
4. עקוב אחר הסטטוס בדשבורד או ברשומות לא תקינות

## טיפול בשגיאות

### סוגי שגיאות

#### 1. שגיאות חיבור

**תסמינים:**
- "לא ניתן להתחבר למקור הנתונים"
- "Connection refused"
- "Authentication failed"

**פתרון:**
1. בדוק את פרטי החיבור (כתובת, נתיב, אישורים)
2. לחץ "בדוק חיבור" בעריכת מקור נתונים
3. ודא שהשרת זמין ונגיש
4. בדוק חומת אש/הרשאות

#### 2. שגיאות אימות

**תסמינים:**
- רשומות מופיעות ב"רשומות לא תקינות"
- הודעה: "Validation failed"

**פתרון:**
1. בדוק את הרשומה ב"רשומות לא תקינות"
2. צפה בשגיאת האימות (איזה שדה כשל)
3. אפשרויות:
   - **ערוך רשומה**: תקן את הערך ושלח מחדש
   - **עדכן Schema**: אם הנתונים תקינים אך Schema מחמיר מדי
   - **התעלם**: סמן כ"התעלם" אם לא רלוונטי

#### 3. שגיאות המרת פורמט

**תסמינים:**
- "Failed to parse file"
- "Unsupported format"
- "Encoding error"

**פתרון:**
1. ודא שפורמט הקובץ תואם להגדרה
2. בדוק קידוד (UTF-8 מומלץ)
3. בעיות CSV: בדוק מפריד, escape characters
4. בעיות Excel: ודא שהגליון הראשון מכיל נתונים

#### 4. שגיאות פלט

**תסמינים:**
- "Failed to write to destination"
- "Topic not found" (Kafka)
- "Permission denied" (Folder)

**פתרון:**
1. ודא שיעד הפלט זמין
2. בדוק הרשאות כתיבה
3. Kafka: ודא ש-Topic קיים
4. תיקייה: ודא שהנתיב קיים והרשאות נכונות

   - **מקור**: מאיזה מקור נתונים
   - **שגיאה**: מה השגיאה (שדה חסר, ערך לא תקין, וכו')
   - **נתונים**: הנתונים המקוריים
   - **תאריך**: מתי התגלתה השגיאה

**פעולות:**
- **צפה**: הצג פרטים מלאים
- **ערוך**: תקן את הנתונים
- **שלח מחדש**: עבד שוב לאחר תיקון
- **התעלם**: סמן כלא רלוונטי
- **מחק**: הסר מהרשימה

## שאלות נפוצות

### כללי

**ש: האם המערכת תומכת בקבצים דחוסים (ZIP)?**
ת: לא בגרסה הנוכחית. יש לחלץ קבצים לפני העיבוד.

**ש: מה הגודל המקסימלי של קובץ?**
ת: ברירת מחדל 100MB. ניתן להגדיר ערך שונה בהגדרות מקור הנתונים.

**ש: כמה זמן נשמרים קבצים מעובדים?**
ת: לפי תקופת השמירה שהוגדרה (ברירת מחדל: 30 יום).

---

### מקורות נתונים

**ש: האם ניתן לעבד אותו קובץ פעמיים?**
ת: לא. המערכת משתמשת ב-deduplication כדי למנוע עיבוד כפול. TTL: 15 דקות.

**ש: מה קורה אם מקור הנתונים לא זמין זמנית?**
ת: המערכת תנסה שוב לפי מדיניות retry (3 ניסיונות, delays: 100ms, 500ms, 1s, 2s, 5s).

**ש: האם ניתן לעבד קבצים היסטוריים?**
ת: כן. הנח קבצים בתיקיית המקור והפעל ידנית (כפתור ⚡).

---

### Schema ואימות

**ש: מה קורה לרשומות שלא עוברות אימות?**
ת: תלוי בהגדרת "פעולה בעת שגיאה":
- **המשך עיבוד**: רשומות תקינות ממשיכות, שגויות נשלחות ל"רשומות לא תקינות"
- **הפסק**: כל הקובץ נדחה

**ש: האם ניתן לשנות Schema אחרי שנוצר?**
ת: כן, אבל זה משפיע רק על קבצים עתידיים.

**ש: מה זה "תבנית Regex"?**
ת: ביטוי רגולרי לאימות פורמט (לדוגמה: `^\d{9}$` לת\"ז). המערכת כוללת עזרת Regex עם תבניות נפוצות.

---

### יעדי פלט

**ש: האם ניתן לשלוח לכמה יעדים בו-זמנית?**
ת: כן! הוסף מספר יעדי פלט במקור הנתונים.

**ש: מה קורה אם יעד אחד נכשל?**
ת: יעדים אחרים ממשיכים. השגיאה נרשמת ב-logs.

**ש: איזה פורמט עדיף לפלט?**
ת:
- **JSON**: עשיר, נפוץ, מומלץ ל-APIs
- **CSV**: פשוט, מתאים ל-Excel/BI tools
- **XML**: legacy systems

---

### תזמון

**ש: האם ניתן לשנות תדירות תזמון?**
ת: כן. ערוך את מקור הנתונים ושנה בכרטיסייה "תזמון".

**ש: מה קורה אם העיבוד קודם עדיין רץ?**
ת: התזמון החדש ימתין לסיום הקודם (no overlapping jobs).

**ש: האם ניתן להשבית תזמון זמנית?**
ת: כן. עבור למקור הנתונים ושנה סטטוס ל"לא פעיל" או שנה תדירות ל"ידני".

## טיפים ומומלצות

### בהתחלת השימוש

1. **התחל עם קטגוריות**: הגדר קטגוריות לפני יצירת מקורות נתונים
2. **השתמש בתבניות Schema**: חוסך זמן ומונע שגיאות
3. **בדוק חיבור**: תמיד בדוק חיבור לפני שמירה
4. **התחל בידני**: צור מקור חדש עם תזמון ידני, בדוק שהכל עובד, ואז הפעל תזמון
5. **התחל קטן**: נסה עם קובץ קטן לפני עיבוד המוני

### אופטימיזציה

1. **השתמש ב-Schema**: אימות מונע נתונים שגויים בהמשך
2. **הגדר תקופת שמירה סבירה**: לא יותר ממה שצריך (חוסך מקום)
3. **סנן קבצים**: השתמש בתבנית שם קובץ כדי לעבד רק קבצים רלוונטיים
4. **מעקב אחר שגיאות**: בדוק "רשומות לא תקינות" בקביעות
5. **השתמש במטריקות**: עקוב אחר דשבורד למניעת בעיות

### מה לא לעשות

❌ **אל תמחק מקור נתונים פעיל בתזמון** (השבת קודם)
❌ **אל תשנה Schema באמצע עיבוד** (המתן לסיום)
❌ **אל תשכח לבדוק חיבור** אחרי שינוי הגדרות
❌ **אל תתעלם משגיאות חוזרות** (בדוק logs ותקן בעיה בשורש)
❌ **אל תגדיר תזמון תכוף מדי** (יעמיס על המערכת ללא צורך)

## מילון מונחים

| מונח | הסבר |
|------|-------|
| **מקור נתונים (DataSource)** | הגדרה של מקום לאיסוף קבצים ממנו |
| **Schema** | מבנה הנתונים הצפוי וכללי אימות |
| **אימות (Validation)** | בדיקה שהנתונים תואמים ל-Schema |
| **רשומה (Record)** | שורה בקובץ או אובייקט JSON |
| **קטגוריה** | סיווג של מקור נתונים לפי תחום עסקי |
| **תזמון (Schedule)** | מתי המערכת תבדוק קבצים חדשים |
| **יעד פלט (Output Destination)** | לאן לשלוח נתונים מעובדים |
| **deduplication** | מניעת עיבוד כפול של אותו קובץ |
| **Cron Expression** | ביטוי לתזמון מתקדם |
| **Pipeline** | שרשרת העיבוד: גילוי → המרה → אימות → פלט |

### נספח א': דוגמאות Schema

**לקוח ישראלי:**
```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["customer_id", "name", "israeli_id"],
  "properties": {
    "customer_id": {
      "type": "string",
      "description": "מספר לקוח"
    },
    "name": {
      "type": "string",
      "minLength": 2,
      "maxLength": 100,
      "description": "שם מלא"
    },
    "israeli_id": {
      "type": "string",
      "pattern": "^\\d{9}$",
      "description": "תעודת זהות (9 ספרות)"
    },
    "phone": {
      "type": "string",
      "pattern": "^0\\d{1,2}-?\\d{7}$",
      "description": "מספר טלפון ישראלי"
    }
  }
}
```

**עסקה פיננסית:**
```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["transaction_id", "amount", "date"],
  "properties": {
    "transaction_id": {
      "type": "string",
      "description": "מספר עסקה"
    },
    "amount": {
      "type": "number",
      "minimum": 0,
      "description": "סכום (ש\"ח)"
    },
    "date": {
      "type": "string",
      "format": "date",
      "description": "תאריך העסקה"
    },
    "type": {
      "type": "string",

      "description": "סוג עסקה"
    }
  }
}
```
